## Description

Generated `AGENTS.md` and `product schema` command (ADR-031). `product agent-init` generates a repo-root file from actual repo state containing: working protocol, current front-matter schemas, domain vocabulary, repository state summary, and MCP tool usage guide. `product schema` returns the complete front-matter schema for any artifact type. Both are exposed as MCP read tools (`product_schema`, `product_agent_context`). Configurable via `[agent-context]` in `product.toml`.

---

## Functional Specification

### Inputs

- `product agent-init [--watch]` — generates or regenerates `AGENTS.md` at the repo root. `--watch` triggers continuous regeneration on front-matter file changes.
- `product schema ARTIFACT_TYPE [--format json] [--all]` — artifact type is one of `feature`, `adr`, `test`, `dep`; `--all` returns all schemas in one document; `--format json` emits JSON Schema.
- `[agent-context]` section in `product.toml` — toggles for `include-repo-state`, `include-schemas`, `include-domains`, `include-tool-guide`; configurable `output-file` path.
- The full knowledge graph (features, ADRs, TCs, dependencies) from the configured paths — used to compute the repo state summary embedded in `AGENTS.md`.
- MCP call `product_schema` with a type argument; MCP call `product_agent_context` (no arguments) to retrieve the generated file content.

### Outputs

- **`product agent-init`** — writes a generated `AGENTS.md` (or the path from `output-file` config) containing: working protocol, current front-matter schemas, domain vocabulary, repository state summary (feature count, phase gate status, ADR count, TC pass/fail counts), MCP tool guide, and a generation timestamp.
- **`product schema ARTIFACT_TYPE`** — human-readable markdown schema reference for the specified artifact type, listing every field with its type, description, and allowed values. `--format json` emits JSON Schema.
- **`product schema --all`** — all four artifact type schemas concatenated in one document.
- **MCP `product_schema`** — identical output to the CLI `schema` command, available over stdio and HTTP transports.
- **MCP `product_agent_context`** — content of the generated `AGENTS.md` file, available programmatically to agents that do not read files directly.
- **Context bundle header** — `product context FT-XXX` includes the `responsibility` field in the ⟦Ω:Bundle⟧ header when set (amendment to ADR-006).

### State

`AGENTS.md` is a generated file written to disk. It is always overwritten on `product agent-init` — hand-edits are not preserved. The generation timestamp is embedded in the file so agents can assess freshness. In `--watch` mode, the file is regenerated on every detected front-matter change; no persistent background process state is retained beyond the watch loop.

### Behaviour

1. **Schema derivation** — `product schema` derives its output from the same Rust type definitions used by the parser (`src/field_schema.rs`). The schema is guaranteed to reflect what Product actually accepts; it cannot drift from the parser's expectations.
2. **`AGENTS.md` generation** — `product agent-init` loads the full graph via `parser::load_all_with_deps`, builds a `KnowledgeGraph`, calls `agent_context::generate_agent_md`, and writes the result atomically via `fileops::write_file_atomic`. The file always starts with a `> Generated by product` line to distinguish it from hand-written files.
3. **Configurable sections** — each section (`include-schemas`, `include-repo-state`, `include-domains`, `include-tool-guide`) can be independently disabled in `[agent-context]` in `product.toml`. Disabling a section omits it from the generated file without error.
4. **Output path** — the default output file is `AGENTS.md` at the repo root. Claude Code reads this automatically at session start. The `output-file` config key redirects output for projects using a different convention (e.g. `.cursorrules`).
5. **`--watch` mode** — uses filesystem notifications to regenerate `AGENTS.md` whenever a file in the features, ADRs, tests, or dependencies directories changes. The watch loop runs until interrupted; each regeneration prints a timestamped notice.
6. **MCP tools** — `product_schema` and `product_agent_context` are MCP read tools exposed over both stdio and HTTP transports. They are available regardless of the `mcp.write` configuration.

### Invariants

- `AGENTS.md` must always contain a `> Generated by product` line. If the file exists without this line, `product agent-init` overwrites it without error — it was hand-edited and is no longer considered authoritative.
- `product schema ARTIFACT_TYPE` output must include every field that the parser accepts for that type. Adding a new parser field without updating the schema output is a bug.
- `product schema --all` must include all four artifact type schemas.
- MCP `product_schema` output must be identical to the CLI `product schema` output for the same artifact type.

### Error handling

- `product agent-init` on a repository with no `product.toml` returns `ProductError::ConfigError` (the config discovery fails before any graph loading occurs).
- Parse errors in individual artifact files are printed to stderr and do not abort generation — the resulting `AGENTS.md` reflects whatever portion of the graph was successfully parsed.
- `product schema UNKNOWN_TYPE` returns `ProductError::ConfigError` naming the unrecognised type.
- MCP `product_agent_context` returns an error if `AGENTS.md` does not exist — callers should run `product agent-init` first.

### Boundaries

- `AGENTS.md` is generated, never hand-edited. Project-specific instructions that should survive regeneration belong in `CLAUDE.md` (hand-written) or another file outside the generated output path.
- `product schema` reflects the parser's type definitions; it does not validate existing artifact files against the schema — that is `product graph check`'s role.
- The `--watch` mode is a development convenience; CI and scripts should call `product agent-init` explicitly rather than relying on the watch loop.
- MCP `product_agent_context` returns the file content as-is; it does not re-generate on every call. Freshness depends on when `product agent-init` was last run.

## Out of scope

- Generating agent instructions for non-Product artifact types or external systems.
- Parsing or executing the instructions in `AGENTS.md` — Product writes the file; the agent framework reads it.
- Automatic commit or push of the generated `AGENTS.md` — the file is written to disk; committing it is the developer's responsibility.
- Schema validation of existing artifact files against the generated schema — use `product graph check` for structural validation.
- Support for agent framework-specific formats beyond plain markdown (e.g. OpenAI assistant files, LangChain tool definitions).
