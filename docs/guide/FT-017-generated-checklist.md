## Overview

The Generated Checklist feature makes `CHECKLIST.md` a read-only view of implementation status rather than a hand-edited source of truth. Status is owned exclusively by the `status` field in each artifact's YAML front-matter. Running `product checklist generate` rebuilds the checklist from the current graph state, ensuring it always reflects reality. This eliminates divergence between front-matter and checklist, a class of bug that is impossible to detect automatically when two sources of truth exist.

## Tutorial

### Generating your first checklist

If you have a repository with feature files already in place, generate the checklist:

```bash
product checklist generate
```

This scans every feature file under the configured `docs/features/` directory, reads the `status` field from each file's YAML front-matter, and writes `CHECKLIST.md` to the repository root.

Open `CHECKLIST.md` and you will see something like:

```markdown
# Implementation Checklist

> Generated by product v0.1 | Last updated: 2026-04-14
> Do not edit directly — update status in feature/test front-matter and run `product checklist generate`

- [x] FT-001 — Cluster Foundation (complete)
- [ ] FT-002 — Graph Queries (in-progress)
- [ ] FT-003 — Context Bundles (planned)
```

### Updating status and regenerating

Change a feature's status using the CLI:

```bash
product status FT-002 complete
```

This updates the `status` field in `docs/features/FT-002-*.md` and regenerates `CHECKLIST.md` automatically. You never need to find and tick a checkbox by hand.

### Verifying the checklist is current

After any implementation work, run:

```bash
product verify FT-002
```

`verify` executes the TC runners linked to the feature, updates test status in front-matter, and regenerates the checklist. If all TCs pass, the feature status moves to `complete`.

## How-to Guide

### Regenerate the checklist after bulk front-matter changes

If you have edited multiple feature files' front-matter directly (e.g., during migration), regenerate in one step:

```bash
product checklist generate
```

The output is idempotent — running it twice on an unchanged graph produces byte-identical output.

### Use the checklist in CI

The checklist itself is informational. For CI gating, use `product graph check`, which returns structured exit codes:

1. Add a CI step:
   ```bash
   product graph check
   ```
2. Interpret the exit code:
   - `0` — graph is clean, no issues
   - `1` — hard errors (broken links, cycles, malformed front-matter) — fail the build
   - `2` — warnings only (orphaned artifacts, missing test criteria)
3. To fail only on errors and tolerate warnings:
   ```bash
   product graph check || [ $? -eq 2 ]
   ```

### Check for status drift before committing

After making changes, confirm the checklist matches front-matter:

1. Run `product checklist generate`
2. Run `git diff CHECKLIST.md`
3. If the diff is non-empty, the checklist was stale — include the regenerated file in your commit

### Migrate from a hand-edited checklist

If your repository has an existing hand-edited `checklist.md`:

1. Run `product migrate` — this reads checked boxes in the existing checklist and populates `status` fields in scaffolded feature files
2. Run `product checklist generate` to produce the generated version
3. Commit the result and stop editing the checklist by hand

## Reference

### Commands

| Command | Description |
|---------|-------------|
| `product checklist generate` | Rebuild `CHECKLIST.md` from current front-matter state |
| `product status <ID> <STATUS>` | Set a feature's status and regenerate the checklist |
| `product verify <ID>` | Run TC runners, update test/feature status, regenerate checklist |
| `product graph check` | Validate graph health (exit codes: 0/1/2) |

### Feature status values

| Status | Checklist marker | Meaning |
|--------|-----------------|---------|
| `planned` | `[ ]` | Not yet started |
| `in-progress` | `[ ]` | Work underway |
| `complete` | `[x]` | All TCs passing, implementation done |
| `abandoned` | n/a | Dropped from scope |

### Exit codes for `product graph check`

| Code | Meaning | CI action |
|------|---------|-----------|
| `0` | Clean graph | Pass |
| `1` | Errors (broken links, cycles, malformed front-matter) | Fail |
| `2` | Warnings only (orphans, missing test criteria) | Pass or fail (team policy) |

### Generated file format

`CHECKLIST.md` always begins with a warning header:

```markdown
# Implementation Checklist

> Generated by product v0.1 | Last updated: <timestamp>
> Do not edit directly — update status in feature/test front-matter and run `product checklist generate`
```

Features are listed with their ID, title, and current status. The file contains no YAML front-matter.

### Configuration

Checklist generation uses paths configured in `product.toml`:

```toml
[paths]
features = "docs/features"
adrs = "docs/adrs"
tests = "docs/tests"
```

## Explanation

### Why the checklist is generated, not hand-edited

The core problem is **source-of-truth divergence**. When status lives in both front-matter and a checklist, they inevitably disagree. Someone updates one and forgets the other. No tooling can reliably detect which is correct because both are equally authoritative.

The solution (ADR-007) is to make front-matter the single owner of status and the checklist a derived view. This means:

- **One place to update**: change status in front-matter (or via `product status`), and the checklist follows
- **Fine-grained audit trail**: `git blame` on a feature file shows exactly who changed its status and when, rather than a single checklist file with many concurrent edits
- **Idempotent regeneration**: `product checklist generate` is a pure function of graph state — running it produces the same output every time for the same inputs, with no accumulated side effects

### Exit code design for CI

ADR-009 defines a three-tier exit code scheme modeled on `grep` and `clippy` conventions:

- **0** = clean
- **1** = errors (must fix)
- **2** = warnings (team decides)

This two-level distinction lets CI pipelines express nuanced policy. A team can block merges on broken links (exit 1) while allowing orphaned artifacts (exit 2) during early development. The exit code is testable directly in shell without parsing stdout — no JSON parsing or custom logic required.

### Relationship to the knowledge graph

The checklist does not store data — it renders it. Every invocation of `product checklist generate` rebuilds the knowledge graph from YAML front-matter (per ADR-003, the graph has no persistent store), reads each feature's status, and writes the checklist. This means the checklist is always consistent with the graph, even if files were edited outside the CLI.

### Relationship to `product verify`

`product verify FT-XXX` is the primary workflow command that ties implementation, testing, and status together. It runs TC runners, updates test status in front-matter, promotes feature status to `complete` when all TCs pass, and regenerates the checklist. The checklist generation step at the end of `verify` ensures developers never need to remember to run `product checklist generate` separately after verification.
