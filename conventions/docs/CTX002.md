# CTX002 — No `.unwrap()` in production code

`.unwrap()` is forbidden in library and binary code. Violations fail
`cargo clippy` with the lint `clippy::unwrap_used` (set to `deny` in
`[workspace.lints.clippy]`).

## Why

`.unwrap()` collapses a recoverable error into a panic. In a CLI that
mutates files on disk and orchestrates LLM tool calls, a panic during an
atomic write or a graph traversal corrupts state and leaves no actionable
message for the user. See [ADR-0002](../adr/ADR-0002-no-unwrap-in-lib.md)
for the full rationale.

## Triggers

- `something.unwrap()` on `Result<T, E>` or `Option<T>` in `src/` or
  `xtask/src/`.
- Method-resolution-equivalent patterns that bypass error propagation
  (e.g. `match opt { Some(x) => x, None => panic!() }`) are not caught by
  this lint and are addressed by code review until promoted to a stricter
  check.

## How to fix

In rough order of preference:

1. **Propagate with `?`.** If the surrounding function returns `Result`,
   replace `.unwrap()` with `?`.
2. **Map to a typed error.** Use `.ok_or(ProductError::Foo)?` or
   `.map_err(|e| ProductError::Bar(e.to_string()))?`.
3. **Provide a default.** `.unwrap_or_default()` or `.unwrap_or(value)` when
   the absence is genuinely benign.
4. **Use `expect("invariant")` deliberately.** Permitted when a constant or
   compile-time-known value cannot fail (e.g. a `Regex::new` over a string
   literal). The string passed to `expect` documents the invariant.

## Examples

**Triggers CTX002:**

```rust
let value: u32 = input.parse().unwrap();
```

**Passes CTX002:**

```rust
let value: u32 = input.parse()?;
// or
let value: u32 = input.parse().map_err(|_| ProductError::InvalidInput)?;
```

## Enforcement

- **Tier:** 3a (Clippy via `[workspace.lints]`).
- **Implementation:** `[workspace.lints.clippy] unwrap_used = "deny"` in
  `Cargo.toml`.
- **Diagnostic code:** `clippy::unwrap_used` (Clippy's own code; no `CTX002`
  prefix because the rule is enforced through Clippy, not xtask).
- **Promotion path:** stays at tier 3a unless we adopt a typed-result
  combinator pattern that makes `.unwrap()` mechanically unrepresentable.
